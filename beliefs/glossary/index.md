# Glossary

One concept file per technical term, alphabetical, maintained by
[`/add-to-glossary`](/.claude/skills/add-to-glossary/SKILL.md). Each definition
is individually linkable — cite a term in any response as a bundle-absolute
link (e.g. `[route tag](/beliefs/glossary/route-tag.md)`) and the link lands on its
definition. Every entry carries a `sense` field marking where its usage lives —
`common` (portable), `repo` (this brain's own vocabulary), or `dual` (both,
defined common-first). See the [glossary hub](/beliefs/glossary.md) for how the
system works.

## Terms

- [abstract syntax tree](/beliefs/glossary/abstract-syntax-tree.md) — The tree-shaped data structure a parser produces from source code — nodes for the program's constructs, stripped of surface syntax — which programs can then inspect, transform, and render back to source.
- [actor model](/beliefs/glossary/actor-model.md) — A concurrency model in which independent, share-nothing processes ("actors") each own their private state and interact only by asynchronous message passing, so contention over a resource becomes a message queue at its owning actor rather than a lock.
- [advisor pattern](/beliefs/glossary/advisor-pattern.md) — An agent cost pattern where a fast, low-cost executor model does the bulk of generation and consults a higher-intelligence advisor model at moments where planning matters, so most tokens bill at the cheaper executor rate while quality approaches advisor-solo.
- [agent memory](/beliefs/glossary/agent-memory.md) — The mechanisms by which an AI agent persists knowledge beyond its context window — spanning short-term, long-term, procedural, and working memory — implemented anywhere from plain markdown files to managed vector databases.
- [agent swarm](/beliefs/glossary/agent-swarm.md) — A multi-agent system in which many concurrently running LLM agents coordinate on shared work, whose aggregate behavior — coordination, cascades, consensus, contention — emerges from the interactions rather than from any single member's competence.
- [agent time horizon](/beliefs/glossary/agent-time-horizon.md) — The human-completion-time length of task an AI agent can finish at a fixed success rate (e.g. the 50% time horizon), used to measure and track agent capability over time.
- [agent-as-presenter](/beliefs/glossary/agent-as-presenter.md) — A workflow inversion in which the agent renders its output inside the human's own tools — navigating their editor to the change, floating rationale beside the code, driving their live browser — instead of into a chat transcript.
- [ambient authority](/beliefs/glossary/ambient-authority.md) — Access a process holds simply by virtue of where and as whom it runs — every same-user resource reachable with no explicit grant — rather than through a capability it was deliberately handed.
- [analysis (type)](/beliefs/glossary/analysis-type.md) — The controlled type for a point-in-time evaluation of a question against evidence, yielding findings and a recommendation, filed under meta/analysis/.
- [append-only](/beliefs/glossary/append-only.md) — A data structure or field that admits new entries but never allows existing ones to be modified or removed — so its history is immutable even as it grows.
- [approval gate](/beliefs/glossary/approval-gate.md) — A control that requires explicit human or permission approval before a tool/MCP action executes; while the approval is unresolved the action is blocked, and it is environment-wide when it blocks every session in an environment.
- [approximate nearest neighbor (ANN)](/beliefs/glossary/approximate-nearest-neighbor.md) — An index/algorithm class that trades exactness for speed when finding the closest vectors to a query in high-dimensional space; its overhead only pays off at large corpus sizes.
- [architecture decision record (ADR)](/beliefs/glossary/architecture-decision-record.md) — A short per-decision document — context, decision, status, consequences — that records an architectural choice and its why at the time it is made, with supersession links chaining records into a de-facto decision history.
- [argumentation framework (Dung)](/beliefs/glossary/argumentation-framework.md) — Dung's abstract model of argument as a directed graph — a set of arguments and a binary attack relation — over which acceptability semantics compute which sets of arguments can rationally stand together.
- [artifact](/beliefs/glossary/artifact.md) — A self-contained, hosted, client-side document (HTML/Markdown/SVG/React) Claude generates alongside a chat reply, rendered as a live page and shareable by URL; canonically defined in the filed concept.
- [assertion graph](/beliefs/glossary/assertion-graph.md) — The pre-OKF knowledge-base design this brain replaced — atomic claims (primitives backed by source evidence, compounds composing primitives via explicit deps) forming a DAG whose remnants live under deprecated/ pending migration.
- [assumption](/beliefs/glossary/assumption.md) — A belief held provisionally rather than on evidence. Senses: (1) TMS — a node whose support depends on the absence of other beliefs, hence retractable; (2) ATMS — a primitive datum from which all others derive; (3) GSN — an intentionally unsubstantiated statement scoped to the whole argument.
- [assurance case](/beliefs/glossary/assurance-case.md) — A structured argument, supported by evidence, that a system satisfies a property (safety, security) in a given context — the regulator-facing deliverable of safety-critical industries, commonly diagrammed in GSN.
- [atomic fact](/beliefs/glossary/atomic-fact.md) — The decomposition unit of factuality evaluation — a short statement carrying a single independently checkable piece of information, extracted from a longer generation so each can be verified against a knowledge source on its own.
- [attestation (belief role)](/beliefs/glossary/attestation.md) — A belief asserted directly by a source or artifact — a primary, atomic belief grounded in what a single source says, as opposed to one derived from other beliefs.
- [attribution](/beliefs/glossary/attribution.md) — In this brain, the frontmatter map recording a doc's ingestion event — when it entered, through which channel and by whom, and why — an immutable record orthogonal to provenance (content origin) and timestamp (last change).
- [attribution shadow](/beliefs/glossary/attribution-shadow.md) — The enforceable trace of an unenforceable runtime act — a choice like model selection cannot itself be gated because the repository never records it, but the committed attribution of who/what performed the act can be required and checked.
- [auto-intake](/beliefs/glossary/auto-intake.md) — The /research step that files its featured items into the bundle via /intake in the same run, moving the operator from a pre-intake gate to a post-intake editorial role.
- [backfill](/beliefs/glossary/backfill.md) — Retroactively populating a newly-added field or dataset for records that predate it, deriving the values from history rather than leaving the past blank.
- [backpressure](/beliefs/glossary/backpressure.md) — Flow control in which a slower downstream stage pushes back on faster producers — through bounded queues, pool limits, or demand signalling — so overload degrades gracefully instead of overwhelming the system.
- [batch invariance](/beliefs/glossary/batch-invariance.md) — The property that an inference server's numerical output for a request does not depend on which other requests it happened to be batched with — the missing ingredient that makes temperature-0 LLM inference actually deterministic.
- [BEAM](/beliefs/glossary/beam.md) — The virtual machine that runs Erlang and Elixir, built around huge numbers of lightweight, heap-isolated processes with preemptive scheduling and message passing — the substrate whose distinctive affordances (supervision, fault isolation, hot code upgrades) target long-running concurrent services.
- [belief network](/beliefs/glossary/belief-network.md) — A probabilistic graphical model — a directed acyclic graph of stochastic variables in which each variable's distribution is conditioned on its parents; also called a Bayesian network.
- [bipartite graph](/beliefs/glossary/bipartite-graph.md) — A graph whose nodes divide into two disjoint sets with edges only between the sets; in belief graphs, belief nodes vs. justification nodes, so one conclusion can carry several independent justifications.
- [blast radius (of a retracted premise)](/beliefs/glossary/blast-radius.md) — The set of beliefs transitively dependent on a premise — everything that needs re-examination when it is retracted or changed; an importance/staleness signal, never a truth score.
- [BM25](/beliefs/glossary/bm25.md) — A lexical full-text ranking function that scores documents against a query by term frequency and rarity, normalized for document length — the standard keyword-search baseline between plain text matching and semantic search.
- [branch transplant](/beliefs/glossary/branch-transplant.md) — Recovering the useful content from a stale unmerged branch by copying its docs onto a fresh branch cut from current main — under the current layout, ids preserved — rather than merging the branch, which would resurrect paths and files the mainline has since moved or deleted.
- [bundle-absolute link](/beliefs/glossary/bundle-absolute-link.md) — A markdown link whose target begins at the bundle root with a leading slash (e.g. /beliefs/glossary/route-tag.md), so it resolves identically from any document in the bundle.
- [candidate feed](/beliefs/glossary/candidate-feed.md) — A regenerated list of external items surfaced as candidates for possible inclusion in a knowledge store, held in a waiting area outside the curated bundle until a human explicitly accepts one.
- [capability-matched model selection](/beliefs/glossary/capability-matched-model-selection.md) — The doctrine that model tier is matched to the epistemic weight of the motion — strongest models where output becomes canonical, a judgment is rendered, or errors are silent; cheaper tiers for derivational, oracle-checked, or high-volume motions.
- [capture problem](/beliefs/glossary/capture-problem.md) — The incentive failure that killed classic design-rationale tools — the person who must author the rationale pays its cost at design time while the benefit accrues to other people later, so humans reliably under-document.
- [CI smoke check](/beliefs/glossary/ci-smoke-check.md) — A lightweight check wired into continuous integration that fails the build if a basic invariant breaks — an early tripwire rather than a full test suite.
- [Claude Agent SDK](/beliefs/glossary/claude-agent-sdk.md) — Anthropic's Python/TypeScript library exposing the same agent engine as Claude Code — the agent loop, tool use, and context management — as an embeddable component you drive from your own process.
- [Claude Managed Agents (CMA)](/beliefs/glossary/claude-managed-agents.md) — Anthropic's hosted agent service (beta) where the agent loop runs on Anthropic's orchestration layer and each session provisions a sandboxed container for tool execution — with versioned agent configs, scheduled deployments, rubric-graded outcomes, and egress-substituted credential vaults.
- [CloudEvents](/beliefs/glossary/cloudevents.md) — A CNCF specification standardizing the envelope of an event — id, source, type, time, data — so events can be produced, routed, and consumed across systems regardless of transport or vendor.
- [code map](/beliefs/glossary/code-map.md) — This brain's generated glossary of its own tooling — meta/code-map.md, compiled from every lib/ module's @moduledoc/@doc/@typedoc by mix brain.codemap so the code that sits alongside the bundle (and carries no em: id) still has a browsable statement of module, function, and type intent.
- [cognitive debt](/beliefs/glossary/cognitive-debt.md) — (Operator-coined) The understanding gap a human operator accrues about a system and its contents — analogous to technical debt — when automation changes the system faster than the operator internalizes it.
- [commoditize your complement](/beliefs/glossary/commoditize-your-complement.md) — A business strategy of making a product complementary to yours cheap, open, or interchangeable, so demand and pricing power concentrate in the layer you own.
- [compiled contract](/beliefs/glossary/compiled-contract.md) — A configuration or policy file treated as build output — regenerated by a compiler task from canonical source documents rather than hand-edited — so there is a single source of truth that cannot silently drift.
- [complementary priors](/beliefs/glossary/complementary-priors.md) — A construction, introduced by Hinton, Osindero & Teh, that cancels the "explaining-away" correlations between hidden causes in a directed belief net, making a deep model's hidden layers conditionally independent and so tractable to learn one layer at a time.
- [Composable Beliefs (cb)](/beliefs/glossary/composable-beliefs.md) — A separate Elixir knowledge/belief-management system built on an OKF-style substrate whose patterns (compiled contract, opaque ids, a route-tag verifier, an evidence/dependency model) were ported and adapted into this brain.
- [computer use](/beliefs/glossary/computer-use.md) — An agent operating a graphical interface the way a human does — reading screenshots and issuing synthetic clicks and keystrokes — the universal but unstructured fallback for software that exposes no API.
- [concept (type)](/beliefs/glossary/concept-type.md) — Controlled type for a definition or mental model — but in practice two speech acts sharing a name (term definition vs accepted proposition), a tension recorded for redefinition.
- [consensus core](/beliefs/glossary/consensus-core.md) — When comparing the belief systems of several artifacts, the subset of beliefs every system supports and none attacks — what survives conflict examination; kin to the grounded extension in argumentation semantics.
- [constant tutorial mode](/beliefs/glossary/constant-tutorial-mode.md) — The reader state of an operator who builds systems through natural-language collaboration and is therefore always learning the system by using it — the audience the plain tier of three-level documentation exists to backstop.
- [Content Security Policy](/beliefs/glossary/content-security-policy.md) — A browser-enforced security standard that restricts which resources a web page may load or connect to, expressed as a policy the page is served with.
- [content-word containment](/beliefs/glossary/content-word-containment.md) — A lightweight lexical-overlap score — the fraction of one text's unique content words (after normalization and stopword removal) that also appear in a reference text — used to flag near-restatements without any semantic model.
- [context mesh](/beliefs/glossary/context-mesh.md) — Loomkin's memory architecture in which overflow conversation context is offloaded at full fidelity to dedicated Keeper processes — with staleness and access-frequency tracking and cross-agent retrieval — instead of being summarized away.
- [context rot](/beliefs/glossary/context-rot.md) — The degradation of an LLM's effective attention and retrieval reliability as its context window fills — memory whose dependability drops with occupancy.
- [contrastive divergence](/beliefs/glossary/contrastive-divergence.md) — A fast approximate training algorithm for restricted Boltzmann machines that updates weights from a short (often single-step) Gibbs-sampling run instead of the intractable exact maximum-likelihood gradient.
- [council round](/beliefs/glossary/council-round.md) — A structured, adversarial, multi-agent review of drafted work, run as comments on a draft PR through four motions (open, review, respond, close), gated on every finding receiving a disposition, and distilled into a curated target document.
- [credential vault](/beliefs/glossary/credential-vault.md) — A store that holds a secret outside the process that uses it and substitutes the real value into outbound requests at egress, so the running (possibly untrusted) code sees only an opaque placeholder — containing the blast radius of prompt injection or code compromise to what the egress boundary allows.
- [CRLF](/beliefs/glossary/crlf.md) — The two-byte carriage-return + line-feed line ending (Windows convention), versus Unix's bare LF; tools that pattern-match on line boundaries must normalize it or honor the file's own ending when writing.
- [cron expression](/beliefs/glossary/cron-expression.md) — A time-based schedule specification — conventionally five fields (minute, hour, day-of-month, month, day-of-week) — that defines when a recurring job fires.
- [cross-reference drift](/beliefs/glossary/cross-reference-drift.md) — The failure mode in which a change that should update many interlinked documents only updates some, leaving the rest silently stale — the known killer of LLM-maintained wikis, and stage 3 of the second-brain failure chain.
- [cross-site scripting (XSS)](/beliefs/glossary/cross-site-scripting.md) — A web vulnerability class in which unescaped attacker-controlled content is interpreted by the browser as markup or script; an unescaped HTML attribute value is one common injection vector.
- [dark factory](/beliefs/glossary/dark-factory.md) — From lights-out manufacturing (a plant so automated it runs with no humans present, hence no lights) — an operation or company run end-to-end by autonomous agents, with humans reduced to governance rather than presence.
- [decision graph](/beliefs/glossary/decision-graph.md) — A persistent DAG of an agent system's reasoning — goals, decisions, options, outcomes as typed nodes and edges — kept so later sessions can recall what was tried, what was rejected, and why; in this brain, the proposed compiled view over capture-time decision records.
- [decision prior](/beliefs/glossary/decision-prior.md) — A proposition held true enough to guide action even when uncertain, subjective, or value-laden — the functional role beliefs play in a knowledge system.
- [decompose-then-verify](/beliefs/glossary/decompose-then-verify.md) — The evaluation pattern of atomizing a long output into individually checkable statements, verifying each against a source, and aggregating the local verdicts mechanically — validated at scale by FActScore and SAFE.
- [decontextualization](/beliefs/glossary/decontextualization.md) — Rewriting an extracted statement to be self-contained — resolving pronouns and implicit scope — because atomic claims ripped from context change or lose meaning; SAFE's revision step, and a required stage of any artifact decomposer.
- [deduplication](/beliefs/glossary/deduplication.md) — Detecting whether a candidate item already exists in a corpus before adding it, to avoid filing duplicates; here performed at intake by searching for semantically equivalent existing concepts.
- [deep belief network](/beliefs/glossary/deep-belief-network.md) — A generative neural network built as a stack of restricted Boltzmann machines, trained greedily one layer at a time, whose 2006 layer-wise learning algorithm helped reignite deep learning.
- [defeasibility](/beliefs/glossary/defeasibility.md) — The property of a claim or inference that holds subject to exceptions — good until defeated by a recognized rebuttal condition, rather than valid absolutely; Toulmin's rebuttal slot builds it into argument structure.
- [dependency-directed backtracking](/beliefs/glossary/dependency-directed-backtracking.md) — On contradiction, tracing the recorded justification structure backwards to the assumptions actually responsible and retracting a culprit — instead of undoing choices in chronological order.
- [dependency-free](/beliefs/glossary/dependency-free.md) — A property of a program that relies only on its language/runtime standard library and pulls in no external packages, so it needs no package-fetch step and runs offline in restricted or ephemeral environments.
- [deploy gating](/beliefs/glossary/deploy-gating.md) — Making a deployment conditional on prior verification passing, so only a validated artifact is published; when a check fails the publish is skipped and the last good deploy stays live (fails safe).
- [description logic](/beliefs/glossary/description-logic.md) — A family of decidable first-order fragments engineered for reasoning over concepts, roles, and individuals — the formal basis of OWL (Direct Semantics aligns with the SROIQ description logic).
- [design rationale](/beliefs/glossary/design-rationale.md) — The reasoning underlying a design choice — the question settled, the alternatives considered, the arguments for and against, and why the chosen one won — studied as a structured representation by the IBIS/QOC/DRL tradition.
- [detector](/beliefs/glossary/detector.md) — A mechanical check that converts a rule violation or drift from a silent state into an announced signal (a failing gate or a printed warning), thereby creating the repair signal that probabilistic enforcement lacks.
- [deterministic spine](/beliefs/glossary/deterministic-spine.md) — The subset of a workflow whose output is fully determined by its inputs — the mechanical tool transforms and checks — as opposed to the non-deterministic, judgment-driven parts; only the spine can be pinned by a conventional test.
- [digest](/beliefs/glossary/digest.md) — Commonly, a compilation that condenses many items into one compact, periodic read; in this brain, the single dated document collecting a day's /research candidate items with synopses, grouped by category.
- [digital twin](/beliefs/glossary/digital-twin.md) — A behavioral clone of an external system — replicating its APIs, state transitions, and edge cases — that stands in for the real thing during testing, so behavior can be exercised at volumes and in failure modes production would never permit.
- [directed acyclic graph](/beliefs/glossary/directed-acyclic-graph.md) — A graph of nodes joined by directed edges that contains no cycles, so following edges never returns to a starting node — the shape of dependencies, pipelines, and orderings.
- [disposition](/beliefs/glossary/disposition.md) — The single verdict a review finding must receive before a review round can converge — agree (state what will change), refute (with evidence, not assertion), or defer (with a named destination where it is filed).
- [distill, don't dump](/beliefs/glossary/distill-dont-dump.md) — The filing rule that a concept captures the knowledge rather than the raw noise — clear title, one-sentence description, clean body — with original material kept as a resource URI or under a Citations section.
- [distillation target](/beliefs/glossary/distillation-target.md) — The curated document that receives a review round's settled outcome — agreed findings as concrete items, rejected alternatives as a dated block — as distinct from a transcript of the exchange.
- [distributed Erlang](/beliefs/glossary/distributed-erlang.md) — The BEAM's built-in clustering — named nodes connect into a mesh and processes exchange messages with PIDs on remote nodes transparently, with `:global` naming and `:pg` process groups — the substrate on which cross-node concurrency and failover are built, but which a framework must deliberately opt into.
- [doctrine](/beliefs/glossary/doctrine.md) — The governance layer of guiding principles and accepted tenets — the "why" that shapes judgment — as distinct from policy, the layer of specific enforceable rules.
- [document (OKF)](/beliefs/glossary/document-okf.md) — The unit of knowledge in this bundle — a single markdown file with YAML frontmatter, whose id is its path minus .md; OKF's own term for it is "concept document".
- [domain-specific language](/beliefs/glossary/domain-specific-language.md) — A small language purpose-built for one problem domain, trading generality for concision and checkability; in Elixir, typically implemented as a macro layer that expands declarations into full code.
- [draft pull request](/beliefs/glossary/draft-pull-request.md) — A GitHub pull request opened in draft state — visible and commentable but explicitly not ready to merge — used to stage review conversation on work in progress.
- [drift class](/beliefs/glossary/drift-class.md) — A category of staleness whose instances share one detection mechanism — e.g. unresolved internal links, or files present in a directory but missing from its index — making the class as a whole checkable by a single detector.
- [DRY (don't repeat yourself)](/beliefs/glossary/dry.md) — The principle that every piece of knowledge should have a single, authoritative representation in a system — duplication creates copies that drift apart, so one of them is eventually wrong.
- [elaboration (type)](/beliefs/glossary/elaboration-type.md) — Controlled type — a persisted expansion of a technical phrase or short passage (quoted target, term definitions, plain walkthrough), written by /elaborate under meta/elaborations/ and back-linked to its originating thread once that session is captured.
- [elixir-mind](/beliefs/glossary/elixir-mind.md) — The ratified new name for this brain itself — the second-brain repository and bundle, per the accepted rename plan (including the sb: → em: id migration) — also used in the 2026-07-14 AST/macros session for the narrower prospective owned-runtime tier of the same project.
- [embeddings](/beliefs/glossary/embeddings.md) — Dense numeric vector representations of text produced by a model such that semantically similar text maps to nearby vectors, enabling similarity comparison via distance metrics like cosine.
- [entailment (logical consequence)](/beliefs/glossary/entailment.md) — The relation "every interpretation satisfying the premises also satisfies the conclusion" — the formal notion a per-edge entailment judgment approximates when a belief graph asks whether premises license a conclusion.
- [entity resolution](/beliefs/glossary/entity-resolution.md) — Deciding when two differently-phrased items refer to the same underlying entity; for belief graphs, unifying equivalently-worded beliefs across documents to a single node — the hard, LLM-mediated core of cross-document comparison.
- [environment (ATMS)](/beliefs/glossary/environment-atms.md) — A set of assumptions, read logically as their conjunction; the unit the ATMS labels beliefs with, and — via its consistent-closure context — the model for one coherent worldview among many.
- [epistemic overlay](/beliefs/glossary/epistemic-overlay.md) — A structural layer above a set of knowledge documents that classifies each by its epistemic role and links them into a network of conceptual dependencies, enabling cross-resource explorability and machine-checkable integrity.
- [epistemic poisoning](/beliefs/glossary/epistemic-poisoning.md) — The contamination of a shared knowledge base with false or attacker-shaped beliefs — typically via an agent ingesting untrusted content and filing it as fact — escalating prompt injection from hijacking one session to corrupting every future reader of the base.
- [epistemic/teleological split](/beliefs/glossary/epistemic-teleological-split.md) — The distinction between statements about what is true (epistemic — beliefs, claims) and statements of standing direction (teleological — doctrine), used as a filing test for where a statement belongs in the type vocabulary.
- [escape rate](/beliefs/glossary/escape-rate.md) — The fraction of defective outputs that pass through a verification gate undetected — the false-negative rate that measures how far a reviewer can trust an automated check.
- [excerpt log (route-tagged log)](/beliefs/glossary/excerpt-log.md) — The generated "## Thread excerpts — route-tagged log" section a route-tag sink carries — per-thread, date-stamped blocks lifting each tagged region whole, re-derivable from the tags and never hand-edited.
- [Execution Context Payload (ECP)](/beliefs/glossary/execution-context-payload.md) — This brain's coined standard for the minimal packet a fresh-context executor receives to start a plan's workstream cold — a thin pointer packet into the ratified plan on disk (plan ref, workstream id, scope + reading allowlist, deliverable contract, verification commands), not the deliberation history that produced the plan.
- [experience graph](/beliefs/glossary/experience-graph.md) — A structured, relational organization of an agent's accumulated successes and failures — in place of ad-hoc reflection or unstructured memory — enabling systematic cross-task reuse of what worked and what didn't.
- [ExUnit](/beliefs/glossary/exunit.md) — Elixir's built-in unit-testing framework, shipped with the standard distribution, supporting fixtures and test tags such as `:tmp_dir` for temp-directory-scoped cases.
- [F1@K](/beliefs/glossary/f1-at-k.md) — SAFE's long-form factuality aggregate — harmonic mean of factual precision (fraction of supplied facts supported) and recall against K, a stipulated facts-per-response target; penalizes both fabrication and under-informing.
- [FActScore](/beliefs/glossary/factscore.md) — Min et al.'s factuality metric (EMNLP 2023) — the percentage of a generation's atomic facts supported by a fixed knowledge source — plus the retrieve-then-verify estimator that automates it to within 2% of human scores.
- [fan-out](/beliefs/glossary/fan-out.md) — Splitting one task into many independent sub-tasks run concurrently, then combining their results — in agent orchestration, dispatching a fleet of subagents to work in parallel (e.g. one reader per subsystem) and synthesizing their structured returns.
- [fast-forward merge](/beliefs/glossary/fast-forward-merge.md) — A git merge where the target branch's tip is an ancestor of the incoming branch, so the pointer simply advances with no new merge commit — leaving the branches identical.
- [fault injection](/beliefs/glossary/fault-injection.md) — A testing technique that deliberately introduces failures — crashed processes, delayed, dropped, or reordered messages, network partitions — into a running system to observe how it degrades and recovers.
- [featuring](/beliefs/glossary/featuring.md) — The gated criteria by which `/research` decides which candidates make the cut: relevance → novelty → a reason tag fires → source quality, under a quality-over-volume cap.
- [feeding pairs](/beliefs/glossary/feeding-pairs.md) — The set of (thread, sink) pairs the current route tags induce — a thread feeds a sink when at least one of its tagged regions lists that sink's stable id; the set is the domain of materialization in both directions.
- [first-order logic (FOL)](/beliefs/glossary/first-order-logic.md) — The standard formal language of objects, predicates, and quantifiers, with Tarskian semantics; sound and complete (Gödel 1929) but only semidecidable — the fully formal pole of knowledge representation.
- [first-parent history](/beliefs/glossary/first-parent-history.md) — The git log traversal (git log --first-parent) that follows only each merge commit's first parent, reading a branch's history as one entry per merge — on a PR-driven default branch, one line per landed pull request.
- [flow (touch-sequence)](/beliefs/glossary/flow-touch-sequence.md) — A documentation artifact that traces a single canonical run of a workflow as an ordered sequence of every file it touches, tagging each step with its actor — distinct from a rules doc, a why-tutorial, or a pass/fail checklist.
- [force-with-lease](/beliefs/glossary/force-with-lease.md) — A safer git force-push (`git push --force-with-lease`) that overwrites the remote branch only if its current tip still matches what the pusher last saw, refusing the push if someone else advanced it in between.
- [fresh-context execution](/beliefs/glossary/fresh-context-execution.md) — Running a bounded unit of work in a newly-spawned agent given only minimal, task-scoped context — the specific inputs it needs — rather than the full accumulated session history.
- [fresh-session-per-fire](/beliefs/glossary/fresh-session-per-fire.md) — A scheduling mode where each firing of a trigger starts a brand-new, clean agent session rather than resuming a persistent one, so a run inherits no state from prior firings.
- [functional core, imperative shell](/beliefs/glossary/functional-core-imperative-shell.md) — A design pattern that isolates pure computation in a stateless core wrapped by a thin imperative layer handling state and effects — the structure an agent instantiates with the model as core and the loop as shell.
- [gate suite](/beliefs/glossary/gate-suite.md) — The full set of blocking verification checks a change must pass before landing — in this brain, compile/format, the tests, and the mix brain.* checks, run manually, by the pre-commit hook, and in CI.

- [graduation](/beliefs/glossary/graduation.md) — A document or work item earning promotion up a level — a glossary term relocating into the domain taxonomy (sense 1), a grounded claim becoming a concept (sense 2), or a deferred strand or analysis residue becoming a plan (sense 3).
- [greedy layer-wise pretraining](/beliefs/glossary/greedy-layer-wise-pretraining.md) — An unsupervised training strategy that builds a deep network one layer at a time — each layer trained on the representation produced by the layer below it — before any end-to-end fine-tuning.

- [id-namespace migration](/beliefs/glossary/id-namespace-migration.md) — A one-time, operator-ratified, tail-preserving rewrite of the id-namespace prefix on every bundle concept — only the prefix changes, each id's immutable 6-hex tail is kept — done in one deterministic, verifier-atomic pass.

- [intent-as-source](/beliefs/glossary/intent-as-source.md) — This brain's doctrine that the human-ratified layer is intent — policies, invariants, specs, tests — with every downstream artifact, code included, a derived and machine-checked product whose opacity must be earned by a mechanical oracle.

- [justification](/beliefs/glossary/justification.md) — The recorded license for holding a belief. - [knowledge graph](/beliefs/glossary/knowledge-graph.md) — A structured representation of entities as nodes and their relationships as typed edges, queryable by traversing edges rather than scanning text; a code knowledge graph applies this to a codebase's files, symbols, calls, and imports.

- [label (ATMS)](/beliefs/glossary/label-atms.md) — The set of minimal consistent environments attached to a belief — every combination of assumptions from which it follows; kept consistent, sound, complete, and minimal by the ATMS, making context membership a subset test.
- [lag-tolerant check](/beliefs/glossary/lag-tolerant-check.md) — A staleness gate for a generated artifact that accepts the on-disk copy being behind its source (missing the newest entries) but fails on any divergence in the content it does contain — used where regeneration structurally trails the data.
- [let-it-crash](/beliefs/glossary/let-it-crash.md) — Erlang's fault-tolerance philosophy — rather than defensively handling every possible error in place, let a failing process crash and have a supervisor restart it in a known-good state, keeping error-handling logic out of business logic.
- [lexical search](/beliefs/glossary/lexical-search.md) — Search that matches on the literal surface form of text — tokens or substrings — rather than meaning; fast and dependency-free, but blind to synonym and jargon gaps.
- [librarian write-broker](/beliefs/glossary/librarian-write-broker.md) — A supervised process that solely owns mutation of a knowledge namespace — agents submit proposed concepts as messages, and the librarian runs the intake gauntlet (dedup, verification, grounding checks) and serializes commits, turning concurrent write contention into a mailbox and making integrity checks unbypassable.
- [lineage](/beliefs/glossary/lineage.md) — The provenance chain that produced an artifact — for a flow doc, the analysis → plan → thread → PR arc — recorded as a canonical frontmatter block from which human-readable views are derived.
- [LLM-as-judge](/beliefs/glossary/llm-as-judge.md) — Using a language model's judgment in place of human annotation for evaluation tasks, validated by measured agreement with human raters — e.g. SAFE matching crowdworkers 72% of the time and winning 76% of disagreements at >20× lower cost.
- [loopback daemon](/beliefs/glossary/loopback-daemon.md) — A background broker process listening on a localhost-only socket, giving processes on the same machine a rendezvous point for structured communication that never leaves the host.
- [LoRA](/beliefs/glossary/lora.md) — Low-Rank Adaptation: a parameter-efficient fine-tuning method that trains two small low-rank matrices modifying a frozen base model's weights, so a specialized variant ships as a megabytes-scale adapter instead of a full model copy.
- [macro](/beliefs/glossary/macro.md) — A compile-time function from code to code — in Elixir, from AST to AST — that lets a small declarative surface expand into full implementation, with errors raised during compilation.

- [markdown renderer](/beliefs/glossary/markdown-renderer.md) — A component that parses markdown source and emits HTML, handling headings, nested lists, tables, fenced code, blockquotes, inline emphasis, and links (including link/anchor rewriting).
- [materialize](/beliefs/glossary/materialize.md) — To compute derived content and write it to disk as ordinary file content, instead of recomputing it on demand — trading always-fresh derivation for inspectability, at the cost of needing a freshness check.
- [maturity model](/beliefs/glossary/maturity-model.md) — A staged framework that describes an organization's or system's progression through discrete capability levels, each with characteristic practices, bottlenecks, and the lever that graduates it to the next.
- [methodology (type)](/beliefs/glossary/methodology-type.md) — The controlled type for a repeatable, prescriptive procedure or playbook — the distilled how-to for a recurring task.

- [mixture-of-experts (MoE)](/beliefs/glossary/mixture-of-experts.md) — A neural-network architecture that sparsely routes each input to a small subset of specialized "expert" sub-networks instead of activating the whole model, giving a large parameter count at a smaller per-token compute cost.
- [model cascade](/beliefs/glossary/model-cascade.md) — A cost strategy that routes each request to the cheapest model likely to handle it and escalates to a more capable, costlier model only on failure or low confidence — trading a small quality risk for large average-cost savings.
- [Model Context Protocol (MCP)](/beliefs/glossary/model-context-protocol.md) — An open client/server protocol standardizing how applications expose tools, resources, and context to LLM agents, so an agent calls a uniform interface instead of each integration being hand-wired.

- [mounting point (harness)](/beliefs/glossary/mounting-point.md) — The file path where a third-party harness expects content to appear (CLAUDE.md at session start, a skill's SKILL.md on invocation) — the harness's convention, distinct from the brain-defined role that fills it.
- [nogood](/beliefs/glossary/nogood.md) — An assumption set discovered to be jointly contradictory, recorded so no context is ever built on it (or any superset of it) again — the ATMS's memory of dead ends.

- [non-monotonic reasoning](/beliefs/glossary/non-monotonic-reasoning.md) — Reasoning in which adding information can remove conclusions — beliefs may rest on the absence of other beliefs (TMS outlists, default rules), so the conclusion set does not grow monotonically with the premise set.
- [observer agent](/beliefs/glossary/observer-agent.md) — An experimental Claude Code subagent role in which a read-only watcher is paired one-to-one with a worker subagent, receives a digest of its activity, and may send at most one advisory, authority-free report — community-reported and absent from official documentation.

- [open weights](/beliefs/glossary/open-weights.md) — A model release mode in which the trained parameters are published for download and self-hosting under a license, without necessarily disclosing training data, code, or recipe — sitting between closed API-only access and fully open-source releases.
- [operating contract](/beliefs/glossary/operating-contract.md) — The root CLAUDE.md that every agent auto-loads at session start, codifying the brain's operative rules — a compiled artifact, not hand-edited.

- [orchestrator pattern](/beliefs/glossary/orchestrator-pattern.md) — A multi-agent cost pattern ("plan big, execute small") where a high-intelligence orchestrator model decomposes a task and dispatches subtasks to multiple cheaper worker models in parallel, so most tokens bill at the lower worker rate.
- [orphan block](/beliefs/glossary/orphan-block.md) — A block of generated output whose source has vanished — here, a dated per-thread block in a sink's excerpt log whose thread no longer tags that sink; the log fidelity check fails on it.
- [orphaned branch](/beliefs/glossary/orphaned-branch.md) — In this brain's git workflow, a remote session branch left lingering after its work concluded — merged-but-undeleted (deletable on sight) or unmerged (never deleted without operator ratification); distinct from git's own parentless orphan branch.

- [probabilistic enforcement](/beliefs/glossary/probabilistic-enforcement.md) — An integrity model in which rules live as prose an agent promises to follow, so each operation has a roughly constant chance of violating them — meaning violations scale with operation count and, absent a detector, compound silently.

- [property-based testing](/beliefs/glossary/property-based-testing.md) — A generative testing technique that asserts invariants ("properties") hold across many automatically generated inputs, rather than over hand-written example cases.
- [proto-belief document](/beliefs/glossary/proto-belief-document.md) — In Composable Beliefs, a durable single page that accretes synthesized knowledge about exactly one matter over time; mapped in this brain onto an OKF concept doc.

- [shared state, dual interfaces](/beliefs/glossary/shared-state-dual-interfaces.md) — An application architecture in which a human-facing UI and an agent-facing structured API operate as two clients of the same live process state — the human gets pixels, the agent gets JSON, and neither scrapes the other.
- [single source of truth](/beliefs/glossary/single-source-of-truth.md) — The design guarantee that a given fact lives in exactly one authoritative place, with every other appearance either generated from it or pointing at it — the only structural (not behavioral) defense against stale copies.
- [single-overview convention](/beliefs/glossary/single-overview-convention.md) — The glossary rule that each term's `description` frontmatter is its one canonical overview — rendered as the entry-page lede and shown verbatim as the index gloss — with the body kept expansion-only, both halves enforced by mix brain.glossary.
- [sink (route-tag sink)](/beliefs/glossary/route-tag-sink.md) — The document a route tag routes into and that accretes the aggregated excerpt log — referenced by a canonical id when it aggregates, or by a path for a non-aggregating back-link.
- [skill](/beliefs/glossary/skill.md) — A named, invocable capability packaged as a SKILL.md under .claude/skills/ (or shipped in a plugin), giving the agent a specialized procedure it runs on request.
- [skill namespacing](/beliefs/glossary/skill-namespacing.md) — Prefixing invocable skill/command names with a group identifier so they resolve as a set; in Claude Code only plugin skills are auto-namespaced, while project skills are invoked bare and collisions resolve by hierarchy.
- [skill-to-skill delegation](/beliefs/glossary/skill-to-skill-delegation.md) — One skill invoking another as a sub-step so the callee's procedure becomes part of the caller's flow, composing behavior instead of duplicating it.
- [software factory](/beliefs/glossary/software-factory.md) — A software organization run as a production line — agents plan, implement, test, and ship while humans supply specifications and acceptance judgment; the AI-era revival of an older term for standardized, tooling-driven software production.
- [soundness and completeness](/beliefs/glossary/soundness-and-completeness.md) — The paired correctness properties of a proof system — sound: everything derivable is valid; complete: everything valid is derivable. Gödel (1929) proved first-order logic has both, aligning syntax with semantics.
- [speculative decoding](/beliefs/glossary/speculative-decoding.md) — An LLM inference technique that proposes tokens with a cheap draft step and verifies them in parallel against the expensive full model, accepting the longest correct prefix — yielding the full model's exact output at higher throughput.
- [spin-out](/beliefs/glossary/spin-out.md) — Extracting a component that grew up inside a larger project into its own independent artifact — its own repository, package, or organization — so it can be versioned, maintained, and consumed on its own, with the original project becoming one consumer among potentially many.
- [spurious (warning)](/beliefs/glossary/spurious-warning.md) — A warning or match triggered by something that is not a real instance of what the check looks for — a false positive; tolerable at warn level but erosive, since repeated spurious output teaches the reader to ignore the channel.
- [squash merge](/beliefs/glossary/squash-merge.md) — A merge that flattens a branch's commits into one brand-new commit on the target, abandoning the originals — linearizing history at the cost of per-commit provenance (messages, trailers, cited SHAs, blame granularity).
- [stable id (em: id)](/beliefs/glossary/stable-id.md) — The opaque, immutable em: + 6-hex-char identifier every bundle concept carries in frontmatter; typed edges reference ids, not paths, so identity survives moves.
- [stale block](/beliefs/glossary/stale-block.md) — A materialized excerpt-log block whose content no longer matches its re-derivation from the current tags — the source changed without re-materializing; caught by the log fidelity check, fixed by re-running --materialize.
- [staleness propagation](/beliefs/glossary/staleness-propagation.md) — Following dependency edges outward when a source or premise changes, so every statement resting on it is flagged for re-verification instead of silently continuing to claim support that no longer holds.
- [statement type](/beliefs/glossary/statement-type.md) — The class of document types (claim, note, concept) that carry agent-authored assertions and so may bear a `verified` field — as opposed to captures and every other type.
- [static-site generator](/beliefs/glossary/static-site-generator.md) — A tool that renders source files — typically markdown with YAML frontmatter — into a complete set of pre-built static HTML/CSS/JS pages, served without any running application backend.
- [strand](/beliefs/glossary/strand.md) — One topic's line of work through a session — the unit a routing ledger tracks, one row per strand, carrying a state (open/paused/closed) orthogonal to where its content was routed.
- [supersession](/beliefs/glossary/supersession.md) — An update model where a revised item records a superseded-by relationship to its predecessor instead of overwriting it, preserving history and enabling staleness to propagate to dependents.
- [synonym expansion](/beliefs/glossary/synonym-expansion.md) — Broadening a search query with alternate phrasings — synonyms, jargon and plain-language variants, acronyms and their expansions — so a lookup finds matches worded differently, bridging vocabulary mismatch without embeddings.
- [teleport](/beliefs/glossary/teleport.md) — Moving a Claude Code cloud session into a local terminal with `claude --teleport`, carrying the session's context so the work continues locally.
- [terminal emulator](/beliefs/glossary/terminal-emulator.md) — The application that owns a terminal window — rendering text from escape sequences and forwarding keystrokes to the programs running inside it — e.g. iTerm2, Ghostty, Terminal.app, or Warp.
- [terminal multiplexer](/beliefs/glossary/terminal-multiplexer.md) — A layer between the terminal emulator and the programs it hosts that multiplexes several terminal sessions into panes and windows, and can inject input into or capture output from any hosted program — tmux being canonical.
- [test oracle](/beliefs/glossary/test-oracle.md) — A source of ground truth, independent of the system under test, that decides whether an output is correct — the component whose independence, not the agent's competence, a verification gate's trust rests on.
- [thread doc](/beliefs/glossary/thread-doc.md) — The frozen, verbatim record of a working session written by /capture under meta/threads/ — narrative, routing ledger, then the tagged User/Assistant render.
- [three-level documentation](/beliefs/glossary/three-level-documentation.md) — An opt-in scheme presenting a document at three tiers — a plain-speak backstop, the canonical terse technical body, and a grounding tier one abstraction below the source — built as one canonical level plus two anchored derivations, never three parallel texts.
- [timestamp](/beliefs/glossary/timestamp.md) — Commonly, a recorded date-time marking when an event occurred or data changed; in this brain, the frontmatter field for a doc's last meaningful change — the semantic modified-date, distinct from attribution.when (created) and git commit dates (mechanical).
- [Tinker](/beliefs/glossary/tinker.md) — Thinking Machines Lab's managed LoRA fine-tuning API (launched 2025-10): low-level Python primitives (forward_backward, optim_step, sample, save_state) over managed distributed-training infrastructure, exporting standard PEFT adapters or merged Hugging Face models.
- [todo (type)](/beliefs/glossary/todo-type.md) — The controlled type for a lightweight actionable task — a single thing to be done, carrying a status (open/done/cancelled), filed under meta/todos/ via the /todo skill.
- [Toulmin model](/beliefs/glossary/toulmin-model.md) — Stephen Toulmin's six-part anatomy of practical argument — claim, grounds, warrant, backing, qualifier, rebuttal — from The Uses of Argument (1958); justification, not formal inference, as argument's first-class object.
- [tower of interpreters](/beliefs/glossary/tower-of-interpreters.md) — The framing of an agent as stacked interpreters, each giving operational meaning to the layer above's output — model over natural language, loop over model emissions, tools over structured arguments.
- [tree is the taxonomy](/beliefs/glossary/tree-is-the-taxonomy.md) — The principle that the directory hierarchy — surfaced level-by-level through index.md files (progressive disclosure) rooted at /index.md — is itself the canonical taxonomy, not a separate maintained map.
- [Tree-sitter](/beliefs/glossary/tree-sitter.md) — An incremental parser-generator and library that builds a concrete syntax tree for source code across many languages, fast enough to reparse on every edit — the parsing layer code-intelligence tools use to extract structure.
- [true merge](/beliefs/glossary/true-merge.md) — A merge that creates a two-parent merge commit wiring the branch's real history into the target's ancestry, keeping every original commit (and its SHA) permanently reachable after the branch is deleted.
- [truth maintenance system (TMS)](/beliefs/glossary/truth-maintenance-system.md) — Classic-AI bookkeeping machinery that records beliefs with the justifications supporting them and revises belief when premises change; justification-based (JTMS, Doyle) and assumption-based (ATMS, de Kleer) variants.
- [TUI (terminal user interface)](/beliefs/glossary/tui.md) — A full-screen interactive program that paints its interface with terminal escape sequences inside whatever terminal emulator it runs in — vim, lazygit, htop, Claude Code — portable anywhere a terminal works.
- [tutorial (type)](/beliefs/glossary/tutorial-type.md) — The controlled type for a long-form explanatory note meant to be read start to finish — the why/how behind the tooling or a topic.
- [two-plane rule](/beliefs/glossary/two-plane-rule.md) — An architectural separation between a fast operational plane (ephemeral, high-frequency state — orders, telemetry, queues — living in databases and process state) and a slow epistemic plane (durable, verified beliefs living in the knowledge bundle), with writes to the epistemic plane kept deliberate and low-frequency by policy.
- [two-tier memory](/beliefs/glossary/two-tier-memory.md) — An agent-memory architecture splitting a fast, private, per-agent working store from a slow, shared, curated canonical store, so agents write freely to their own tier and promote validated knowledge into the shared tier through a gatekeeper.
- [typed edge](/beliefs/glossary/typed-edge.md) — A machine-traversable, id-based relationship declared in a document's structured metadata, reserved for relationships tooling must consume or enforce — as opposed to an untyped prose link.
- [ungrounded inference](/beliefs/glossary/ungrounded-inference.md) — A derived belief that does not resolve, transitively through its justifications, to any evidence leaf (attestation/source) — the primary defect an epistemic-graph integrity check flags.
- [vector database](/beliefs/glossary/vector-database.md) — A store that indexes items as high-dimensional embedding vectors and answers nearest-neighbor queries by vector similarity, enabling semantic rather than lexical retrieval.
- [verification grounding](/beliefs/glossary/verification-grounding.md) — The rule that a `verified: true` agent statement must be backed by evidence — a non-empty `verified_by` pointing at captures — and never by its own `resource` link.
- [verified_by](/beliefs/glossary/verified-by.md) — The frontmatter field holding a statement's evidence edges — an inline YAML list of stable ids that jointly support it, required non-empty for verified true.
- [vibe architecting](/beliefs/glossary/vibe-architecting.md) — Architecture shaped by prompts rather than deliberate design — AI coding agents selecting frameworks, scaffolding infrastructure, and wiring integrations in seconds, with almost no one reviewing those choices as architectural decisions.
- [von Neumann architecture](/beliefs/glossary/von-neumann-architecture.md) — The classic stored-program computer organization — processing unit, control unit, memory, and I/O — which the four-component agent decomposition (model, loop, memory, tools) reproduces.
- [wake-sleep algorithm](/beliefs/glossary/wake-sleep-algorithm.md) — An unsupervised training procedure for layered generative networks that alternates a "wake" phase (adjust generative weights from data-driven recognition) with a "sleep" phase (adjust recognition weights from the model's own generated samples).
- [warn pass](/beliefs/glossary/warn-pass.md) — An advisory check that reports its findings without failing — the signal is a printed warning rather than a red gate — used where detection is mechanical but the underlying rule is tolerant or the judgment ultimately editorial.
- [warrant](/beliefs/glossary/warrant.md) — In the Toulmin model, the statement authorizing the movement from grounds to claim — the usually implicit license that makes facts count as support for this conclusion; the thing a per-edge entailment judgment actually evaluates.
- [webhook](/beliefs/glossary/webhook.md) — An HTTP callback that inverts polling — instead of a client repeatedly asking a service whether something changed, the service POSTs to an endpoint the client operates the moment an event occurs, requiring an always-on receiver that authenticates and reacts to each delivery.
- [well-founded support](/beliefs/glossary/well-founded-support.md) — The anti-circularity condition on belief in a TMS — every held belief must trace to premises through non-circular arguments; mutually supporting beliefs with no external grounding don't count.
- [Workflow (orchestration tool)](/beliefs/glossary/workflow-tool.md) — The Claude Code execution environment's orchestration primitive — a script that fans work out to fresh-context subagents (agent(), with parallel()/pipeline() for concurrency, phase() for grouping, and optional git-worktree isolation) and returns synthesized results to a thin orchestrating session.
- [write-time vs build-time](/beliefs/glossary/write-time-vs-build-time.md) — This brain's placement rule for LLM-generated content in a deterministic pipeline — produce it at write-time (the authoring agent commits it as reviewable source) and never at build-time (inside a derivation step, where non-determinism breaks byte-compare checks, demands CI secrets, and destroys reproducibility).
