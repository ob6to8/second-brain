# Plan 003: Assertion DAG Knowledge Architecture

## Context

The assertion graph currently stores 6 notes as monolithic markdown files (articles, videos, conversations) organized by content type. This works for accumulation but not composition — notes are isolated artifacts with no mechanism for beliefs to build on each other, get updated by new evidence, or surface contradictions.

The user's assertion DAG thesis demonstrates that explicit dependency graphs between atomic claims produce dramatically better knowledge composition than prose. This plan restructures the vault from a note repository into a belief system with provenance, where sources are evidence, assertions are composable claims, and an index encodes the DAG.

Design principles from conversation:
- **Shell-first.** Scripts over skills, skills over natural language.
- **Agent-agnostic.** Data format and scripts work without Claude Code. Skills are a convenience layer.
- **No Obsidian assumptions.** Markdown + wikilinks stay (text conventions), but no design decisions optimize for Obsidian.
- **Built for scale.** Architecture handles thousands of assertions. Current corpus is seed.
- **Assertion DAG as knowledge model.** Sources are evidence. Primitives are atomic claims. Compounds compose primitives with explicit implications.

## Architecture

```
Layer 0: Files on disk (markdown, json, shell scripts)
Layer 1: Shell scripts (ingest, build-index, query, deps, orphans, contested)
Layer 2: Claude Code skills (/search, /conclude — thin wrappers over Layer 1)
Layer 3: Natural language conversation (research, reasoning)
```

### Directory structure (target)

```
assertion-graph/
├── sources/                    # External content (articles, videos, papers, journals)
├── assertions/                 # Primitives and compounds (distinguished by frontmatter)
├── schema/                     # System rules — file specs, conventions, index format
├── templates/                  # Frontmatter templates for source + assertion
├── scripts/                    # Shell tooling
├── plans/                      # Architecture plans (historical)
├── intake/                     # Staging area
├── .claude/
│   └── skills/                 # Claude Code skills (/search, /conclude)
├── CLAUDE.md                   # Agent instructions (references schema/)
└── index.json                  # Generated DAG index
```

Flat within `sources/` and `assertions/` — no topic subdirectories. Tags handle taxonomy.

## Steps

### Step 1: Create schema files

Create `schema/` directory with four files defining the system rules.

**`schema/conventions.md`**
- Filenames: kebab-case, derived from title or claim
- IDs: same as filename (no extension)
- Tags: lowercase, hyphenated
- Wikilinks: `[[id]]` format (no path prefix, no extension)
- Dates: ISO 8601 (YYYY-MM-DD)
- Summaries/claims: single sentence or short phrase, written for scanning

**`schema/source.md`**
- Required fields: `type: source`, `title`, `tags`, `date_captured`
- Optional fields: `url`, `aliases`, `source_type` (article|video|paper|journal|transcript), `authors`, `channel`, `date_published`, `summary`, `duration`, `word_count`
- `related`: wikilinks to other sources or assertions this connects to
- Body: extracted/curated content

**`schema/assertion.md`**
- Required fields: `type: assertion`, `tier` (primitive|compound), `claim`, `tags`, `date_asserted`
- `claim`: single sentence, the atomic belief
- `confidence`: high|medium|low|contested
- `evidence`: array of objects, each with:
  - `source`: wikilink to a source or inline context string
  - `type`: citation|supporting|reasoning|axiom
- Compound-only fields:
  - `deps`: array of wikilinks to prerequisite assertions
  - `implication`: string — what the combination of deps produces (the compositional insight)
- `supersedes`: wikilink to an older assertion this replaces (for belief evolution)
- Body: optional elaboration, reasoning, or context beyond the claim

**`schema/index.md`**
- `index.json` structure: `{ sources: [...], assertions: [...], edges: [...] }`
- Source entries: id, type, source_type, title, tags, file
- Assertion entries: id, type, tier, claim, confidence, tags, deps (compounds only), file
- Edge entries: `{ from, to, type }` where type is `evidence|dependency|supersedes`
- Generated by `scripts/build-index.sh` — files are source of truth, index is derived cache

### Step 4: Rewrite `scripts/build-index.sh`

Update to generate the new index format:
- Scan `sources/*.md` and `assertions/*.md`
- Extract frontmatter via `yq`
- Build sources array, assertions array, and edges array
- Edges derived from: `evidence[].source` wikilinks → evidence edges, `deps[]` wikilinks → dependency edges, `supersedes` → supersedes edges
- Output as `{ sources: [...], assertions: [...], edges: [...] }`
- Validate: warn on broken wikilinks (referenced ID doesn't exist)

### Step 5: Create new CLI scripts

**`scripts/query.sh`** — search the knowledge base
```bash
# Search by tag
./scripts/query.sh --tag markdown
# Search by claim text (grep through assertion claims)
./scripts/query.sh --claim "markdown"
# Search by type
./scripts/query.sh --type assertion --tier compound
# Full text search across all files
./scripts/query.sh --text "vector search"
```
Implementation: `jq` queries against `index.json`, falls back to `rg` for full-text.

**`scripts/deps.sh <assertion-id>`** — walk the DAG upward
- Given an assertion ID, print its dependency tree (deps, and their deps, recursively)
- Also show evidence sources at leaf nodes
- Output as indented tree

**`scripts/orphans.sh`** — find unconnected nodes
- Assertions with no dependents (nothing depends on them) and no compounds referencing them
- Sources with no evidence links from any assertion
- Helps identify knowledge that hasn't been integrated

**`scripts/contested.sh`** — find assertions needing attention
- Assertions with `confidence: contested`
- Assertions with `confidence: low`
- Could extend to detect contradictions (two assertions with conflicting claims on same tags)

### Step 6: Update `CLAUDE.md`

Rewrite to reflect new architecture:
- Reference `schema/` for file specs instead of inline documentation
- Document the two content types (source, assertion) and their relationship
- Document the layered architecture (shell → skills → natural language)
- Update ingestion workflow: URL → source file, then optionally extract assertions
- Add assertion workflow: research/conversation → decompose → create/update primitives → compose compounds
- Remove Obsidian-specific language
- Remove old directory references (articles/, videos/, etc.)
- Keep script documentation, dependency list

### Step 7: Create `/conclude` skill

**`.claude/skills/conclude/SKILL.md`**

The skill reads the conversation context and:
1. Identifies distinct claims/findings from the thread
2. Reads `index.json` to find existing assertions that overlap
3. For each new claim:
   - If an existing assertion covers it: update evidence, adjust confidence
   - If genuinely new: create a primitive assertion file
4. Identifies compositions — where multiple primitives imply a compound
5. Creates compound assertion files with explicit deps and implication
6. Ingests any source URLs referenced as source files (if not already present)
7. Runs `./scripts/build-index.sh`
8. Commits all changes

Allowed tools: `Read, Write, Edit, Glob, Grep, Bash, WebFetch`

Output: lists what was created/updated, shows the new edges in the DAG.

- Update output format: "Related assertions" section in addition to "Related notes"

Update `MEMORY.md` to reflect the architectural shift — new directory structure, assertion DAG model, schema-driven conventions.

## Relationship to Plan 002

Plan 002 (persistent conversation architecture with `llm` CLI) is orthogonal to this work. The `llm` CLI provides a persistence mechanism for conversation turns. This plan restructures what the *deliverable* of a conversation is (assertions, not monolithic notes). They can be implemented independently. When Plan 002 is executed, `export-conversation.sh` would call `/conclude` logic rather than writing a single conversation file.
