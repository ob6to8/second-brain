# Why `meta/dev-history.md` drifted, and why the fix isn't a CI auto-commit bot

**Question.** CI on `main` failed (`ElixirMind.DevHistoryTest`) after PR #132 merged,
with `DevHistory.check() == {:stale, "meta/dev-history.md"}`. Given the file is
*derived* — generated by `mix brain.dev_history` from the git merge graph — how can a
derived file drift, and if it needs a fix, why can't that fix just be a CI hook that
regenerates and commits it automatically, triggered by the merge itself rather than by
any particular PR?

## Evidence 1: "derived" means generated-once-and-committed, not computed live

`meta/dev-history.md` follows the same discipline as `CLAUDE.md`, `meta/registry.md`,
and `meta/code-map.md`: a mix task renders a snapshot from source-of-truth state (here,
`git log --first-parent` on `main`) and the snapshot is committed like any other file.
Nothing re-renders it when new commits land later — it reflects the merge graph as of
whenever someone last ran `mix brain.dev_history`.

`lib/elixir_mind/dev_history.ex` documents the resulting constraint directly (module
doc, `check/1`): a PR's branch can never contain its own merge commit — GitHub creates
that commit at merge time — so the checked-in file can be at best one PR behind a fresh
derivation. `check/1` is therefore **lag-tolerant by exactly one PR**
(`lagging_but_consistent?/2`: the fresh render must equal the disk copy plus zero or
more *additional* trailing sections).

## Evidence 2: the actual failure was more than one PR of lag

The GitHub Actions run history on `main` showed CI failing across three consecutive
merges (PR #130, #131, #132) before the bookmarks-reorganization PR tripped it. That means the checked-in file fell behind by three PRs, not one — past what
`lagging_but_consistent?/2` tolerates. Nobody had run `mix brain.dev_history` in any of
those three PRs to catch the file back up to "as of the previous merge," which is the
one thing each of them *could* have done. This is a process gap, not a flaw in the
one-PR-lag design itself.

Separately, `pages.yml` (the GitHub Pages deploy workflow) already regenerates the file
live before building the site on every push to `main`, so the *published* page is
always current regardless of the checked-in copy's staleness. The bug only affects the
git-tracked snapshot and the CI gate over it, not what a reader sees on the live site.

## Evidence 3: a CI-bot auto-commit, fired on the merge and outside the PR, was considered

The natural next question: since `pages.yml` already runs `mix brain.dev_history` on
`push: branches: [main]`, why not add a step there (or a sibling workflow) that diffs
the regenerated file against disk and, if stale, commits and pushes the fix straight to
`main` — decoupled entirely from any PR? Technically nothing blocks this; it's a
standard pattern (lockfile bots, `terraform fmt` bots).

Two things make it a real departure from how this bundle is built, not just a stylistic
preference:

1. **It breaks the PR-only-landing convention on `main`.** Every write to `main` here
   goes branch → PR → merge (git-workflow policy; the `create-pull-request` flow's own
   invariant: "never commit to a default branch"). A bot pushing directly to `main`
   after the fact is a commit with no branch, no PR, and no review — the exact pattern
   the rest of the workflow exists to avoid, applied specifically to the branch that
   has the least tolerance for it.
2. **It has no session to attribute to.** The merge-strategy policy treats the commit
   graph as a provenance layer precisely because every commit traces to an agent
   session (the `Claude-Session:` trailer, or the PR body as fallback anchor). A
   scheduled Actions job is not a session — there is no transcript, no trailer to
   write. The policy already enumerates the known trailer-less cases (pre-feature-era
   commits, GitHub's auto-generated merge-button commits, local-terminal sessions); a
   bot auto-commit would be a genuinely new, uncatalogued category, and adopting it
   would mean amending that policy, not just adding a workflow step.

There's also a structural ceiling worth naming: even a bot firing right on the merge
can't make the file perfectly self-consistent. A commit can't contain a rendering of
history that includes its own hash — the content has to exist before it can be
committed. So a merge-triggered bot only shrinks the lag from "until the next PR" to
"until the bot's own follow-up commit"; it doesn't eliminate the self-referential lag,
it just moves the goalposts closer. The one-PR-lag design already accepts a version of
this ceiling; a bot buys a smaller number, not a categorically different guarantee.

## Recommendation

Fold `mix brain.dev_history` into the `/create-pull-request` skill's commit step,
matching how the other generated-but-committed docs are already kept fresh:
`CLAUDE.md`, `meta/registry.md`, and `meta/code-map.md` are regenerated *inside* the
authoring session, before that session's own commit — never patched after the fact by
something outside the PR. This keeps every write to `main` attributable to a session
and closes the drift gap in practice, since essentially all traffic into this repo is
already Claude-session-authored PRs. It's not a hard guarantee the way the other three
docs' **non**-lag-tolerant `--check` gates are (a PR opened outside this skill would
still need someone to remember), but it matches the one-PR-lag design's own accepted
ceiling rather than trying to buy a false sense of completeness with a bot commit that
the provenance model has no place for.

See [`meta/flows/create-pull-request.md`](/meta/flows/create-pull-request.md) for the
resulting pipeline step, and the
[merge-strategy policy](/meta/policy/merge-strategy.md) for the provenance model this
recommendation preserves.
