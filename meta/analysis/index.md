# Analysis

Point-in-time evaluations and decision-support write-ups — a question investigated
against evidence (often the live bundle itself), yielding findings and a
recommendation, filed so the reasoning and its conclusion persist. A separate
namespace from `plans` (intended *work* to execute) and `issues` (problems to
track): an analysis is a *reasoned judgment on a question*.

Each analysis is a `type: analysis` doc.

## Contents

- [Should the council-round protocol be integrated into this repo?](/meta/analysis/council-round-suitability.md) — yes, as a `/council` skill after four ratified bindings: the adversarial draft-PR review protocol composes with the bundle's existing machinery (true-merge provenance supplies its cite-by-SHA guarantee, persist-plans its hand-off artifact, the verifier culture a path to mechanizing its disposition gate) and fills the undefined drafted→merged gate — but its no-transcripts rule must be narrowed to defer to session-capture, its distillation target bound to `meta/plans/`, and its draft-PR open reconciled with `/create-pull-request`.
- [Is this brain an operational substrate for the engineer-as-orchestrator role?](/meta/analysis/alignment-with-the-orchestrator-role.md) — evaluates the bundle against the four orchestration domains in Anthropic's 2026 coding-trends role description (architecture design, agent coordination, quality evaluation, strategic decomposition); finds the brain's operating loop already practices the role (compiled contract, ratification, gated evaluation, multi-strand triage) and its distinctive contribution is judgment-context across delegated work, while ranking the gaps: broken unattended delegation, unmeasured semantic quality, serial-only session coordination, no standing work-in-flight view, and a missing doctrine anchor for "the right problems."
- [Should the brain add a wiki, and where do its docs fall short?](/meta/analysis/docs-surface-evaluation-and-wiki-question.md) — a full-surface docs audit (all gates green) recommending against a GitHub wiki — a second content store outside the enforcement stack, whose role the verified Pages site already fills — and locating the real gaps: a stale README, two index omissions, one broken policy-example link, and tutorial/flow coverage holes, tiered by action.
- [Would CB's four-typed DAG, as an overlay, stabilize this knowledge base against the failure chain?](/meta/analysis/cb-epistemic-overlay-as-failure-chain-stabilizer.md) — yes for cross-reference drift and trust collapse (both through supersession + staleness propagation), no for the dedup entry gate, conditional elsewhere; portable as a format, but enforcement and edge-authoring must stay host-native. Includes a source-staleness check that corrected two claims traced to CB's outdated thesis doc — itself a live demonstration of the drift failure mode.
- [Is the corpus-maintenance failure space rich content for evals?](/meta/analysis/eval-suitability-of-the-corpus-maintenance-failure-space.md) — yes: ground truth is constructible because the operator owns the corpus; maps each failure-chain stage to a measurable eval (dedup recall, fan-out completeness, filing consistency, capacity curves, longitudinal corpus-health) and decomposes the seven sources of fuzziness in "is this a duplicate" ground truth, each dictating a gold-set property.
- [How does this bundle compare to the 2026 second-brain field?](/meta/analysis/comparison-with-the-2026-second-brain-field.md) — a landscape survey (Obsidian/PARA influencer systems, Karpathy-pattern Claude Code vaults, PKM products) finds no public system combining this bundle's enforcement stack; models the five-stage failure chain that kills advisory systems past ~500 concepts, and names intake dedup recall as the one semantic gate this bundle's CI can't cover.
- [Belief decomposition over artifacts: derive the graph, never author it](/meta/analysis/belief-decomposition-derived-vs-authored.md) — evaluates the atomic-belief-DAG idea against its own two-failure lineage (assertion graph, Composable Beliefs) and the prior art (TMS, Toulmin, decompose-then-verify, GSN, FOL/OWL); finds a derived, regenerable overlay escapes the authored-store failure mode and recommends the analysis-mode plan.
- [Would a vector DB improve recall as this bundle scales?](/meta/analysis/vector-db-recall-for-the-scaling-bundle.md) — a dedup-recall probe over the live corpus finds grep already misses existing concepts on natural-phrasing (semantic, not typographic) queries; recommends synonym-expanded intake dedup + a repeatable recall probe over any standalone vector DB.
- [Is "contract" a necessary abstraction, and what family does it belong to?](/meta/analysis/contracts-and-rendered-aggregations.md) — inventories the bundle's five rendered aggregations and finds the contract a deliberate singleton distinguished by authority (it alone binds agents); "contract" names a role decoupled from the `CLAUDE.md` mounting point, "Agent-contract" adds no contrast, and skills join the pattern as sources (frontmatter → contract §7), not targets.
- [Would deploying to the BEAM — or integrating Jido 2 — benefit this brain?](/meta/analysis/beam-deployment-and-jido-2-evaluation.md) — finds the Elixir layer is a deliberately zero-dependency batch toolchain whose real runtime is Claude Code sessions plus CI, so a resident BEAM deployment buys nothing today and Jido 2 is triply blocked (toolchain floor, zero-dep constraint, duplicate agent runtime); maps a tiered future path (resident daemon → MCP tool server → Jido agents) with concrete adoption triggers, the nearest being the broken /research Routine and tier-2 embedding dedup.
- [If this brain became the epistemic base for a dark-factory agent network, does the BEAM/Jido case expand?](/meta/analysis/dark-factory-epistemic-base-beam-jido.md) — yes, qualitatively: a lights-out agent company inverts the "runtime is elsewhere" premise, dissolving two of the three blockers and reducing the third to a migration cost, making the BEAM near-canonical substrate (supervision as the night-shift manager, a librarian write-broker, taxonomy as signal-topic space); the hard problem moves to epistemic governance under concurrent autonomous writers, whose prerequisites are exactly the active plans (dedup probe, epistemic overlay) — the critical path is unchanged, only the stakes rise.
- [When to roll your own advisor-pattern harness instead of Anthropic's native advisor tool](/meta/analysis/when-to-roll-your-own-advisor-harness.md) — weighs Anthropic's native server-side advisor tool (cheap executor consults a stronger advisor mid-task) against a self-built two-call orchestration; the native tool wins on convenience (single request, built-in pause/resume, per-iteration cost accounting) while rolling your own buys seven controls (advice visibility, advisor system prompt, advisor context, call timing, multi-critic ensembles, model-pairing freedom, GA-only stability) that turn decisive only under named constraints — auditability, multi-vendor reach, data residency, existing-orchestration integration, GA-only policy, product-defining advisor behavior, multi-critic needs — with cross-vendor mixing a facet of the multi-vendor case, not a reason of its own; recommends the native tool by default and separates the choice from mixture-of-experts (a model-internal architecture, not an application-layer pattern).
- [Do the second-brain → second-mind taxonomies point anywhere this bundle should head?](/meta/analysis/second-mind-taxonomy-and-the-belief-gap.md) — evaluates an external ChatGPT dialogue's layered taxonomies (seeded with this bundle's own doctrine definition) against the live bundle: the six-layer stack converges on the existing type vocabulary and the bundle already externalizes epistemology/ontology/ethics/teleology with stronger enforcement; rejects confidence-scored beliefs, a values type, and capability-ladder types; isolates one real gap — `/beliefs/` holds no beliefs and operator-held value-laden priors have no type — resolved in the [belief-layer plan](/meta/plans/belief-type-and-beliefs-namespace.md).
- [Managed minds, owned machinery: Claude Managed Agents vs. Jido/BEAM for an agent-operated knowledge system](/meta/analysis/claude-managed-agents-vs-beam-jido.md) — a first-principles division of labor: agentic work splits into deliberative (rare, long, stochastic) and mechanical (constant, sub-second, mostly deterministic) kinds, and a framework fits the kind whose granularity matches its unit of agency — CMA's unit is a rented Claude session (deliberative plane), Jido's a supervised process that need not call an LLM (mechanical plane); the planes meet at one owned write-gatekeeper, enforced by reducer invariants in-process and credential custody over the network, and the epistemic layer (dedup, grounding, staleness) precedes both substrates.
