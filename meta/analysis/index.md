# Analysis

Point-in-time evaluations and decision-support write-ups — a question investigated
against evidence (often the live bundle itself), yielding findings and a
recommendation, filed so the reasoning and its conclusion persist. A separate
namespace from `plans` (intended *work* to execute) and `issues` (problems to
track): an analysis is a *reasoned judgment on a question*.

Each analysis is a `type: analysis` doc.

- [Is "contract" a necessary abstraction, and what family does it belong to?](/meta/analysis/contracts-and-rendered-aggregations.md) — inventories the bundle's five rendered aggregations and finds the contract a deliberate singleton distinguished by authority (it alone binds agents); "contract" names a role decoupled from the `CLAUDE.md` mounting point, "Agent-contract" adds no contrast, and skills join the pattern as sources (frontmatter → contract §7), not targets.
- [Would deploying to the BEAM — or integrating Jido 2 — benefit this brain?](/meta/analysis/beam-deployment-and-jido-2-evaluation.md) — finds the Elixir layer is a deliberately zero-dependency batch toolchain whose real runtime is Claude Code sessions plus CI, so a resident BEAM deployment buys nothing today and Jido 2 is triply blocked (toolchain floor, zero-dep constraint, duplicate agent runtime); maps a tiered future path (resident daemon → MCP tool server → Jido agents) with concrete adoption triggers, the nearest being the broken /news Routine and tier-2 embedding dedup.
- [If this brain became the epistemic base for a dark-factory agent network, does the BEAM/Jido case expand?](/meta/analysis/dark-factory-epistemic-base-beam-jido.md) — yes, qualitatively: a lights-out agent company inverts the "runtime is elsewhere" premise, dissolving two of the three blockers and reducing the third to a migration cost, making the BEAM near-canonical substrate (supervision as the night-shift manager, a librarian write-broker, taxonomy as signal-topic space); the hard problem moves to epistemic governance under concurrent autonomous writers, whose prerequisites are exactly the active plans (dedup probe, epistemic overlay) — the critical path is unchanged, only the stakes rise.
- [Claude Managed Agents vs. the BEAM/Jido direction: alternative, tandem, or both?](/meta/analysis/claude-managed-agents-vs-beam-jido.md) — the comparison dissolves along the two-plane line: CMA (beta) rents the slow-plane worker runtime (supervision, audited scheduled deployments, versioned agents, vaults, outcomes, a native ratification primitive) while its unit of agency — always a Claude session — misfits the fast-plane machinery (staleness cascades, dedup indexing, verification sweeps), where Jido's non-LLM agents, in-process librarian integration, and pure-reducer harness (testable, replayable, provably gated) are genuine advantages; the librarian sits at the border, and the /news trigger gains a managed structural fix.
