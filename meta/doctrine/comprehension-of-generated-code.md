# The operator must be able to reconstruct a mental model of code they did not write

This is a **standing direction** — the intention the brain's session-capture,
routing-ledger, and route-tagging machinery instrumentally serves, stated so the
*why* survives independent of any one mechanism that implements it. It is not an
enforceable rule (that is what a [`policy`](/meta/policy/index.md) is); it is the
direction plans, policies, and priorities may cite as the end they serve.

## The direction

When a human hand-writes software, comprehension is **not optional and not
separate** — it is fused to production. Progress cannot occur without a mental
model, because you cannot type the next line without holding the shape of the
system in your head. Hand-coding *forces* the model into existence as a side
effect of building.

Agent generation severs that coupling. Code can now ship correct and complete
with **zero operator comprehension** — the model is no longer a precondition of
the work, so it no longer forms for free. When code is, as in the operator's
case, effectively 100% generated, comprehension becomes a **separate, deliberate
act** that either the process produces or nobody does. There is no longer a
forcing function; the brain must *be* the forcing function.

This doctrine states the resulting mandate: **the operator must be able to
reconstruct a mental model of software they did not write, and the brain's design
must actively produce that model — it will not appear on its own.**

## What it commits the brain to

- **The capture→routing→tagging apparatus exists to serve comprehension, not
  archival.** [`/capture`](/meta/policy/session-capture.md) freezes the linear
  session; the [routing ledger](/meta/policy/routing-ledger.md) and
  [route tags](/meta/policy/route-tagging.md) excise topical spikes from that
  linear record and anchor them to the document or code they concern. The point
  of that reorganization is precisely to convert an immutable progressive record
  of events (which no human holds as a mental model) into a topical map (which
  they do). This doctrine is the reason that machinery is worth its cost.
- **Comprehension is both spatial and causal, and the design must serve both.**
  A usable mental model of generated code is two models at once: *what this is*
  (the spatial map of the system) and *why this and not the alternative we
  rejected* (the causal/temporal narrative of how it came to be). Co-locating
  explanation with code serves the first; preserving the linear thread's
  through-line serves the second. A design that serves only one starves the model
  it needs whole.
- **The mental model is the system's only persistent worldview, and it has two
  readers.** Agents are stateless across sessions — each rehydrates from the
  immutable record, the code, and the contract; there is no resident agent
  worldview. So the brain's documents-and-tags *are* the persistent mental model,
  read by the operator (spatial cognition, next session) **and** by the next
  cold-start agent (which must query it to reconstruct). Comprehension
  infrastructure serves both; a form that serves the human's cognition but not the
  agent's reconstruction (or vice versa) is half-built.
- **Presentation is a legitimate axis of comprehension, separate from storage.**
  How the operator *reads* the model — a chat scroll-back, a doc-side log, an
  in-editor overlay driven into their own tools — is a first-class design concern,
  not cosmetics, because comprehension forms at the site of reading. The
  [agent-as-presenter direction](/meta/analysis/agent-drivable-apps-shared-state-dual-interfaces.md)
  is this doctrine applied to the delivery surface.

## Relationship to the orchestrator doctrine

This is the **comprehension precondition** of
[the engineer as orchestrator](/meta/doctrine/engineer-as-orchestrator.md). That
doctrine charges the human with *quality evaluation* of delegated output and with
*ensuring the system solves the right problems* — both of which presuppose that
the human comprehends what the agents produced. The orchestrator doctrine names
*what the human does*; this one names *the comprehension without which they cannot
do it*. Orchestration at broad scope is only real if the scope is understood, and
understanding of generated code is exactly what no longer comes for free.

## Standing against this direction

The concrete question of *where and how* development conversation should live so
that it best produces the operator's mental model — linear thread projected to
topical sinks, versus conversation co-located with the artifact it concerns — is
worked through in
[Should development conversation be stored co-located with the code it concerns?](/meta/analysis/localized-code-conversation-vs-linear-thread.md).
That analysis is a design judgment scored against this direction; this doctrine is
the fixed intention it serves.
