# Tutorials

Long-form explanatory notes about how the brain's tooling and governance work —
the "why", written to be read start to finish. Distinct from `policy` (the rules
that compile into the contract) and `threads` (verbatim conversation archives).

## Contents

- [The tooling architecture — a grand tour of the `mix brain.*` pipeline](/meta/tutorials/the-tooling-architecture.md) — the spine that ties the other tutorials together: how the `elixir_mind` modules form a parse → scan → generate → publish pipeline over the bundle's markdown, with a cross-cutting provenance layer and a thin `mix brain.*` CLI on top; where each narrow tutorial and the generated [code map](/meta/code-map.md) fit into that shape.
- [Why the brain's toolchain runs offline in CI and any sandbox](/meta/tutorials/why-the-toolchain-runs-offline.md) — how zero external dependencies (`deps: []`) let every `mix brain.*` task, including the site generator, build with no network, and the CI/sandbox specifics that make that matter.
- [What makes a commit show as "Verified" on GitHub](/meta/tutorials/what-makes-a-commit-verified-on-github.md) — author vs committer identities, why "Verified" is a signature check (not an email), why re-authoring an agent commit to `noreply@anthropic.com` lets this environment's signature attach, and the only-rewrite-unpushed-commits rule.
- [Bundle scope and non-bundle namespaces](/meta/tutorials/bundle-scope-and-non-bundle-namespaces.md) — why the brain has four independent scanners (identity/verify, route-tags, contract, site), how the registry's exclusion list *defines* what a bundle concept is (and why adding `inbox/` there is what keeps candidate feeds from breaking CI), and why the site deliberately renders namespaces the registry ignores.
- [The three bundle scanners — Registry, Verifier, and RouteTags](/meta/tutorials/the-three-bundle-scanners.md) — one crawler (`Registry.scan/1`: wildcard + directory-exclusion + frontmatter parse) and two consumers; RouteTags adds a second surface over `meta/threads` and joins the governance and knowledge namespaces by stable id; why excluded dirs (e.g. `test/` fixtures) are invisible to all three.
- [The gate suite — what the checks prove and where they run](/meta/tutorials/the-gate-suite-and-where-it-runs.md) — the eight CI gates in four kinds (build, generated-artifact freshness `--check`, bundle validation `verify`/`route_tags`, site render), what turns each red and how to fix it, and the three surfaces they run on (an agent's scoped manual pass, the opt-in pre-commit hook, and the authoritative CI job).
- [How the Pages deploy is gated on a verified bundle](/meta/tutorials/gating-the-pages-deploy-on-a-verified-bundle.md) — why GitHub Actions' per-workflow isolation means a `needs:` can't reach across from `pages.yml` to `ci.yml`, and how repeating the integrity checks in the Pages build job makes the live site reachable only for a bundle that verifies.
- [The session-init digest — how open work finds a fresh session](/meta/tutorials/the-session-init-digest.md) — how `mix brain.session_init` compiles open issues, todos, active plans, and dangling ledger strands into the digest the SessionStart hook injects; the heuristic top-3 class weights, the strand↔doc dedup, the operator's `priority:` escape hatch, and why the script ranks while the agent judges.
- [Why a true merge (not a squash) keeps cited commits reachable](/meta/tutorials/why-a-true-merge-keeps-cited-commits-reachable.md) — reachability as the mental model: deleting a branch deletes a ref, not commits; a squash-merge abandons the original SHAs while a true merge wires them into `main`'s ancestry, so documents that cite commits by SHA (like a plan citing port commits) stay valid after the branch is deleted.
- [The agent's branch lifecycle across merged PRs](/meta/tutorials/branch-lifecycle-across-merged-prs.md) — why a session pins one designated branch name, how a merged PR forces that branch to restart from the default branch under the same name (a new PR each cycle, never a reopen), and why auto-deleting the merged head branch makes the restart a clean create instead of a force-push over a stale tip.
- [The git workflow — single-trunk, true-merge, gate-before-deploy](/meta/tutorials/the-git-workflow.md) — the end-to-end version-control loop: one durable trunk (`main`) that *is* the production version, a short-lived `claude/<slug>` session branch per unit of work in an isolated worktree, a local pre-commit gate then an authoritative CI gate on every push and PR (presubmit + `on: push` postsubmit), a true merge that preserves session→commit provenance, head-branch deletion, and continuous deployment to the live Pages site gated by re-verification — plus where it sits against trunk-based development and the two-tier reference it was audited against, and the open hardening directions (pre-commit auto-wiring, generated-artifact merge handling, review depth).
- [Sampling and quantization as a model of software testing](/meta/tutorials/sampling-and-quantization-as-a-model-of-testing.md) — a signal-processing reading of what a test suite does to a program's behavior: tests *sample* a continuous behavior-signal at discrete inputs (coverage is sample density; inter-sample peaks are worst cases in the gaps) and *quantize* each reading to a finite precision (assertion strength is bit depth; property-based inputs are dither), with invariants as the Nyquist band-limit — and the one place the analogy breaks, that code has no band-limit so no sample rate guarantees reconstruction. The full form of the sampling frame in the [compilation-target analysis](/meta/analysis/code-as-natural-language-compilation-target.md).
